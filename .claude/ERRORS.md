# 📋 Инструкция по использованию

## Когда добавлять ошибку
1. **Технические ошибки** — ReferenceError, TypeError, синтаксис
2. **Процессные ошибки** — забыл Read перед Write, неправильный формат TaskUpdate
3. **Архитектурные** — неправильный импорт, нарушение паттерна

## Как добавлять
Используй точный формат (см. шаблон в `.claude/templates/ERROR_ENTRY.md`):
```
**Дата:** YYYY-MM-DD
**Ошибка:** Краткое название
**Причина:** Root cause analysis
**Исправление:** Как было решено
**Не повторять:** Конкретный паттерн для избежания
```

## Когда НЕ добавлять
- Обычные исправления опечаток
- Успешно завершенные задачи (это в git history)
- Бизнес-логические изменения

## Проверка при старте сессии
Я автоматически проверяю этот файл в начале сессии и избегаю известных ошибок.

## Примеры по категориям

### Tool-Related
- Edit: не найдена строка для замены
- Write: файл не был прочитан перед редакцией
- Read: слишком большой файл без offset/limit

### Process-Related
- TaskUpdate: неправильный формат статуса
- Git workflow: неправильный формат ветки
- Build: непроходящий npm run build

### Code-Related
- Import: забытые импорты (особенно interpolate)
- Types: неправильное использование TypeScript
- CMS: нарушение draft-state pattern
