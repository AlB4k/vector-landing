# Глобальные инструкции Claude Code

## Architecture decisions
- **CMS Draft-State Pattern**: Все изменения в CMS сначала попадают в локальный стейт («черновик») и применяются к основному контенту только после нажатия кнопки «Сохранить». Это предотвращает случайную порчу данных на живом сайте.
- **interpolate() Utility**: Использование централизованной утилиты для замены плейсхолдеров в строках. Весь текст на фронтенде должен проходить через эту функцию для поддержки динамических данных из CMS.
- **CMS-First Approach**: Конфигурация интерфейса (включение секций, темы, лейблы) управляется через JSON-контент в CMS. Прямой хардкод параметров в компонентах запрещен.

## Экономия токенов
- Читай только те файлы, которые нужны для текущей задачи
- Не читай весь проект при старте — жди конкретного запроса
- Не повторяй содержимое файлов в ответе — только изменения
- Не пиши длинных объяснений если не просят — код + краткий комментарий
- Предпочитай точечные правки (str_replace) полной перезаписи файлов
- Не запускай инструменты "на всякий случай" — только если необходимо
- MCP-серверы используй только когда задача явно требует их возможностей

## Уточнение задачи
- Если задача неоднозначна — задавай вопросы по одному, не все сразу
- К каждому вопросу давай краткое пояснение зачем он важен
- Если возможно — предлагай 2-3 варианта ответа на выбор
- После получения ответа задавай следующий вопрос с учётом предыдущего
- Не приступай к работе пока не прояснены все ключевые неоднозначности

## Память об ошибках
- При старте сессии читай .claude/ERRORS.md если файл существует
- При обнаружении любой ошибки — сразу добавляй запись в .claude/ERRORS.md
- Файл создаётся автоматически hook-скриптом, тебе создавать его не нужно
- Формат записи:
  **Дата:** YYYY-MM-DD
  **Ошибка:** что пошло не так
  **Причина:** почему это произошло
  **Исправление:** как было решено
  **Не повторять:** конкретный паттерн которого надо избегать
- Никогда не повторяй подход который уже зафиксирован в ERRORS.md
- Если собираешься сделать что-то похожее на зафиксированную ошибку — предупреди

## Дизайн и фронтенд
- Перед любым UI-кодом загружай skill frontend-design
- Выбирай конкретное эстетическое направление ДО написания кода
- Никогда не используй Inter, Roboto, Arial, system-ui как основной шрифт
- Никогда не делай фиолетовые градиенты на белом фоне
- Каждый проект должен иметь уникальную визуальную идентичность

## Язык
- Всегда общайся на русском если явно не указано иное
- В коде — комментарии на русском

## Работа с большими файлами
- Никогда не читай файл целиком если нужна только часть — используй grep/head/sed
- Перед чтением файла >200 строк спроси какая именно секция нужна
- При разрастании контекста предупреди и предложи выполнить /compact
